先从Socket在iOS项目中的应用谈起，谈到iOS中的Socket，用的较多的是CocoaAsyncSocket，在此基础上，使用Socket更加便捷。
Socket通常用在需要服务端给客户端实时发送消息的场景，比如股票的实时行情，比如实时的消息。那早期的聊天消息，实际上是基于UDP的，鉴于早期网路基础设施不发达，每秒几kb的速度，UDP的传输效率更高，现在的聊天，比如微信是基于TCP，QQ基于UDP及腾讯的私有协议（就是基于WebScoket或者Scoket原生进行封装的一个聊天协议），Socket又是TCP的媒介。顺便提一下，当前的聊天软件的语音聊天和视频聊天是基于RTP（Real-time Transport Protocol, 传递音频和视频的标准数据包格式）的，通常会是RTP和RTP的控制协议RTCP协作。

使用Socket可能遇到的问题:

1.粘包或窜包
2.多线程操作导致Socket的连接异常

先说第一种导致粘包或窜包的原因：
TCP是基于Socket的通信的，TCP是一种流式传输协议（stream protocol）。这就意味着数据是以字节流的形式传递给接收者的，没有固有的”报文”或”报文边界”的概念。从这方面来说，读取TCP数据就像从串行端口读取数据一样，无法预先得知在一次指定的读调用中会返回多少字节（也就是说能知道总共要读多少，但是不知道具体某一次读多少）。
如以下场景：

  • 从A主机向B主机推送两条消息D1和D2
  • 理想的传输是
  • 实际的传输过程可能是

这就可能造成粘包或窜包。
一般针对粘包的分包（或称拆包）的方法是：给每个数据包添加头部，头部中包含数据包的长度，这样接收到数据后，通过读取头部的长度字段，便知道每一个数据包的实际长度了，再根据长度去读取指定长度的数据便能获取到正确的数据了。
完整的数据包 = 服务号 + 数据长度 + 数据
数据包头 = ld(4B) + length(4B) 共占用8个字节
数据包 = length(假设100个字节)
所以这条消息的长度是108字节，要先知道一条数据的边界，只要知道了数据的length字段即可。
